# プロジェクト開発ルール (CLAUDE.md)

## 🚨 最重要・絶対遵守事項
**作業開始前に必ず `AI_GUIDELINES.md` を読み、そのルールを「AI 自身の憲法」として遵守すること。**
新機能の実装や修正よりも、「既存の正常な状態を 1 ミリも壊さないこと」を最優先する。

## 1. 修正の手順（鉄の掟）
1.  **現状の完全把握**: 修正前に必ず `git diff` を実行し、現在の安定状態を確認する。
2.  **ピンポイント置換**: `replace_file_content` を使い、必要な行だけを最小限に修正する。ファイル全体の書き換えは「厳禁」。
3.  **余計なことをしない**: インデントの修正、未使用変数の削除、コメントの書き換えなど、指示されていない変更はバグの元になるため一切行わない。
4.  **単一ファイル・単一修正**: 一度の指示で複数のファイルを修正したり、複数の目的を同時に達成しようとしたりしない。
5.  **BOM/文字化けの徹底排除**: 日本語が含まれるファイルは必ず「BOM なし UTF-8」で扱う。PowerShell パイプ経由の書き出し（`>` や `Out-File`）は文字化けを引き起こすため禁止し、Git の直接操作（`checkout` 等）を優先する。

## 2. 検証とデプロイ
- **セルフチェック**: 修正直後に `git diff` で「指示していない箇所の変更」がないか、AI 自身が 1 行ずつ確認報告する。
- **逐次コミット**: 修正が 1 つ完了するたびに、具体的な内容と共にコミットし、セーブポイントを作る。
- **Netlify デプロイ**: Site ID を直接指定して行う（`netlify deploy --site 63a3902a-8d74-4914-9d80-2e5cf53a28d8 --dir d:\契約 --prod`）。
- **即時ロールバック**: デプロイ後に少しでも挙動が怪しい、またはバグ報告があった場合は、迷わず直前のコミットまで `git checkout` または `git revert` で戻す。

## 3. システム整合性の維持
- **送信・受取の同期**: `sign-editor.js` と `signing.html` の見た目・挙動を常に一致させる。
- **描画タイミング**: ドキュメント読み込み（PDF/画像）完了後に必ず描画関数を呼び出す。
- **データ保護**: ローカル保存データの構造変更時は後方互換性を維持する。
- **エラー通知**: 通信エラー時はユーザーに具体的対策を提示する UI を作る。

## 4. モバイル・レスポンシブ開発
- **メディアクエリの分離**: スマホ専用の調整は必ず `@media` クエリ内に閉じ込め、PC 版の共通スタイル（`.field-marker` 等）を直接壊さない。
- **マルチデバイス検証**: 修正後は PC (1280px+) とスマホ (390px-) 両方の表示を確認すること。
- **ネイティブ化の意識**: 将来のアプリ化を見据え、タッチしやすいサイズ（44px+）を確保しつつ、PC での間延びを防ぐ。

## 5. 本番環境の保護
- 既存の API レスポンス形式や DB スキーマは絶対に破壊しない。
- 環境変数（`.env` 等）の設定変更は極めて慎重に行い、本番の `NODE_ENV=production` を上書きしない。
- 本番 URL（diffsense.spacegleam.co.jp）での動作確認を怠らない。

---
**このルールは、ユーザーからの個別のリマインドがなくても、AI が自律的に実行・監視・遵守しなければならない。**
