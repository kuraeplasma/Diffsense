# AI 開発安全ガイドライン (AI_GUIDELINES.md)

このプロジェクトを改修する AI は、以下のルールを**絶対的に遵守**し、機能の追加よりも「既存機能の破壊（デグレード）」を防ぐことを最優先すること。

## 1. 最小限の変更原則 (Minimal Change Principle)
- **部分置換の徹底**: ファイル全体を書き換えるのではなく、指示された箇所のみを `replace_file_content` でピンポイントに修正すること。
- **無関係な変更の禁止**: インデントの微調整、空行の追加・削除、変数名の勝手なリファクタリングなどは、バグの元になるため「一切禁止」とする。
- **コメントの維持**: 既存のコメントを削除したり、書き換えたりしないこと。

## 2. 破壊的変更の防止 (Prevention of Regressions)
- **修正前の状態確認**: 修正前に必ず `git diff` や `git log` で現在の安定した状態を把握すること。
- **変更後の自己検閲**: 変更を適用した後、デプロイ前に必ず `git diff` を実行し、「指示していない箇所に意図しない変更が入っていないか」を 1 行ずつセルフチェックすること。
- **エンコーディングの厳守**: 日本語が含まれるファイルは必ず「BOM なし UTF-8」で保存すること。PowerShell のパイプなど、文字化けの恐れがある操作は一切禁止する。

## 3. 段階的な適用と検証 (Incremental Deployment)
- **一回一ファイル一機能**: 複数の問題を一度に直そうとしない。1つの修正が完了し、デプロイ後の安定が確認されるまで、次の修正には着手しない。
- **デプロイメッセージの具体化**: 何を修正したかを具体的に Git コミットメッセージに含め、後から原因を特定できるようにすること。
- **本番前プリフライト強制**: 本番デプロイ前に必ず `npm run preflight:prod` を実行し、失敗時はデプロイを中止すること。
- **配信先固定**: `diffsense.spacegleam.co.jp` への本番反映は Netlify Site ID `63a3902a-8d74-4914-9d80-2e5cf53a28d8` を使用し、Firebase Hosting を誤って使わないこと。

## 4. DIFFsense 専用・不整合防止ルール (System Integrity)
- **エディター・受取側の同期 (Editor-Recipient Parity)**: 署名枠やバッジのスタイル、座標ロジックを修正する際は、必ず `js/sign-editor.js`（送信側）と `signing.html`（受取側）の両方をセットで確認し、見た目と挙動を完全に一致させること。片方だけの修正は「不完全」とみなす。
- **非同期描画の確実な実行**: PDF や画像（fallback）の読み込みは非同期である。ドキュメントの読み込み完了（Promise の解決）を待ってから、必ず描画関数（`renderFields` / `renderFieldMarkers`）を呼び出すこと。
- **データスキーマの保護**: IndexedDB や localStorage に保存されるデータの構造を変更する場合は、既存の保存データが壊れないよう、必ず後方互換性を維持するコードを含めること。
- **親切なエラーハンドリング**: API 通信（404/500等）が発生した際は、`console.error` で終わらせず、ユーザーに状況と対策を日本語で伝え、再試行可能な UI を提供すること。

## 5. モバイル・レスポンシブ開発の安全ルール (Mobile First with PC Protection)
- **PC レイアウトの死守**: スマホ対応のために PC 版のレイアウトやフォントサイズを「直接」変更することは禁止。必ず `@media (max-width: ...)` クエリ内でモバイル用の差分のみを記述すること。
- **!important 汚染の防止**: モバイル用に `!important` を安易に追加しない。PC 版に意図しないスタイルが波及するのを防ぐため、詳細度やクラス設計で解決すること。
- **両画面での検証**: スマホ UI を修正した後は、必ず「PC 解像度（1280px以上）」でも崩れがないか目視で確認し、報告すること。将来のネイティブ化を意識したタッチターゲット（44px以上）の確保を推奨するが、PC のクリック精度を犠牲にしない。

## 5. エラー発生時の即時ロールバック
- **疑わしきは戻す**: 修正後に新しいバグ報告があった場合は、原因究明を急ぐ前に、まずは最後に安定していたコミットへ即座に `git checkout` または `git revert` で戻し、サービスを保護すること。

---
**このガイドラインは、ユーザーが個別に指示しなくても、AI が作業を開始するたびに自動的に適用されるものとする。**
